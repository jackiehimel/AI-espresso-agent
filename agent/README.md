# AI Espresso

A daily AI news digest generated by a multi-role LLM agent. Each edition contains three story cards and one prompt-of-the-day card, rendered as HTML + Markdown and published to the AI Garage portal.

## Quick start

```bash
cd agent
pip install -r requirements.txt
export ANTHROPIC_API_KEY=sk-ant-...

# 1. Generate today's edition JSON (agent run)
python3 -c "import datetime as dt, espresso_agent; \
  espresso_agent.run(dt.date.today(), dry_run=False, use_cache=True, mode='agent')"

# 2. Render to HTML + Markdown + illustrations (optional: --no-images)
python3 render_edition.py $(date +%Y-%m-%d)

# 3. Preview in browser (agent → render → local server; Ctrl+C to stop)
python3 preview_edition.py $(date +%Y-%m-%d) --use-cache
python3 preview_edition.py 2026-05-18 --render-only   # re-render + open existing JSON
python3 preview_edition.py 2026-05-18 --no-images     # faster, broken image icons OK
python3 preview_edition.py 2026-05-18 --write-archive # opt in to archive mutation
```

`preview_edition.py` skips archive writes by default (sets `ESPRESSO_SKIP_ARCHIVE=1` during preview runs) so local previews do not mutate cross-edition memory.

Output:
- `data/editions/YYYY-MM-DD.json` — raw edition data + agent trace
- `../editions/edition_N_variant_c.html` — rendered email-ready HTML
- `../editions/edition_N_variant_c.md` — same content as Markdown
- `../editions/edition_N/assets/variant_c_NN.png` — four illustrations

The `editions/` directory at the repo root is the contract surface that the AI Garage portal consumes via its existing `sync-espresso.mjs` script.

### Dev backfill (multiple dates)

There is no `run_chain.py` in this repo. To backfill several consecutive editions locally (so each day's `archive.jsonl` entry feeds the next run's dedupe), loop `espresso_agent.run` yourself:

```bash
cd agent
for d in 2026-05-16 2026-05-17 2026-05-18; do
  python3 -c "import datetime as dt, espresso_agent; \
    espresso_agent.run(dt.date.fromisoformat('$d'), dry_run=False, use_cache=True, mode='agent')"
  python3 render_edition.py "$d"
done
```

Not used by CI — production uses `.github/workflows/daily-edition.yml` only.

## Architecture

```
┌─────────┐   shortlist + gaps   ┌──────────────────────────────┐
│  Scout  │ ───────────────────▶ │ Editor (native tool_use loop) │
└─────────┘                      │  pick · search_news · memory  │
                                 │  self_critique · ship_edition │
                                 └──────────────────────────────┘
            │
            ▼
   edition JSON
            │
            ▼
   render_edition.py
            │
            ▼
   variant_c HTML + MD + 4 PNGs
            │
            ▼
   committed to editions/
            │
            ▼
   AI Garage sync picks it up
```

### Scout (`espresso_loop.py` — `SCOUT_SYSTEM`, `run_scout()`)

Pulls candidate stories from a 45-source catalog (see `sources.yaml`) organized into four tiers:

| Tier | Examples |
|---|---|
| T1 — Frontier press & primary | Anthropic, OpenAI, DeepMind, NYT Tech, The Verge, TechCrunch, Bloomberg, arXiv cs.AI |
| T2 — Expert commentary | Stratechery, Latent Space, Simon Willison, Pragmatic Engineer, Ethan Mollick |
| T3 — Newsletters | TLDR AI, Rundown AI, Superhuman |
| T4 — Industry verticals | MIT Tech Review, FierceHealthcare, STAT |

Scout scores candidates on freshness, source tier, and topical fit, then hands the Editor a shortlist of ~14 stories along with explicit coverage gaps. The Editor uses these gaps to decide when to call `search_news`.

### `search_news` backends

Production (CI and local on Python 3.11) should use **`PERPLEXITY_API_KEY`** — the [Perplexity Search API](https://docs.perplexity.ai/docs/search/quickstart) (`POST /search`) returns `results`, normalized to the same hit shape the agent expects (`title`, `url`, `snippet`, `domain`) before allow-list filtering.

| Backend | When | Requirement |
|---|---|---|
| **Search API** (recommended) | `PERPLEXITY_API_KEY` or `PPLX_API_KEY` set | Python 3.11+, `httpx` (already in `requirements.txt`) |
| **pplx CLI** (optional) | `pplx` on `PATH` and CLI succeeds | `pip install pplx-cli` on **Python 3.12+** only |

Order: try `pplx search web` first if the binary exists; on failure or absence, call the API. If both are unavailable, the tool returns `{"error": "search failed — …"}` (never silent) and the trace records it in `result_summary` so the Editor can note search was unavailable.

### Editor (`espresso_loop.py` — `EDITOR_SYSTEM`, `run_tool_agent()`)

Runs an Anthropic native `tool_use` loop. The model chooses tools and order; Python dispatches and enforces ship gates. Tools:

| Tool | Purpose |
|---|---|
| `pick(slot, id, reason)` | Assign a candidate to a slot |
| `unpick(slot)` | Remove the current pick from a slot |
| `read_candidate(id)` | Read the full body of a shortlist item |
| `search_news(query)` | Web search beyond the daily fetch (max 2 per edition). See **search_news backends** above — prefer `PERPLEXITY_API_KEY` in production |
| `check_archive(headline)` | Fuzzy dedupe against recent editions |
| `update_memory(key, value)` | Write working memory (pool quality, gaps, notes) |
| `note_weak_pool(reason, adjustments)` | Document a thin news day (required for 3-story weak-pool editions) |
| `self_critique()` | Rubric review — returns `approve` or `revise` |
| `ship_edition()` | Terminal tool when gates pass |

### Ship gates

`ship_edition` succeeds only when: required picks are filled (4 normally, 3 on documented weak-pool days), tier-1 minimum is met, vendor cap respected, constitution checks pass (`constitution.py`), and the last `self_critique` was `approve`. If the Critic approves but `ship_edition` is not called (budget exhaust), the run **salvages** the approved slate (`mode: agent`, warning in notes). With zero recoverable picks, the run writes `data/editions/YYYY-MM-DD.failed.json` and fails CI unless `ESPRESSO_ALLOW_DETERMINISTIC_FALLBACK=1` (local dev only).

### Controller (`espresso_loop.py` — `agentic_select()`)

Scout bootstrap → `run_tool_agent()` (soft budget 25, hard budget 40 tool calls). Records every event to `agent_trace` including `tool_use_id`. On unrecoverable agent failure, the run writes `data/editions/YYYY-MM-DD.failed.json` and **raises** (CI fails). It does **not** silently fall back unless you opt in locally (see below).

### Deterministic fallback (local dev emergency only)

Production and `.github/workflows/daily-edition.yml` always use `mode="agent"`. The legacy `rank_and_select` path (`RANKING_SYSTEM` in `espresso_agent.py`) is **deprecated** and is **not** kept in sync with `_EDITORIAL_RUBRIC` in `espresso_loop.py` — do not rely on it for editorial quality.

| Knob | Purpose |
|---|---|
| `mode="agent"` (default) | Scout → Editor → Critic native `tool_use` loop |
| `mode="deterministic"` | Skip the agent loop; run `rank_and_select` directly (legacy) |
| `ESPRESSO_ALLOW_DETERMINISTIC_FALLBACK=1` | After agent failure with no recoverable slate, run `rank_and_select` instead of failing |
| `ESPRESSO_SKIP_ARCHIVE=1` | Dev/local only: write edition JSON but skip `append_archive` in `write_edition` (does not replace Phase 1.1 upsert-by-date) |

Use the env flag or `--mode deterministic` only on your machine when debugging source fetch or ranking plumbing — never in CI, cron, or client-facing runs. Example:

```bash
# Agent failed; inspect trace first — fallback is last resort:
ESPRESSO_ALLOW_DETERMINISTIC_FALLBACK=1 python3 espresso_agent.py --date 2026-05-18 --mode agent
# Local agent run without mutating archive.jsonl:
ESPRESSO_SKIP_ARCHIVE=1 python3 espresso_agent.py --date 2026-05-19 --use-cache --mode agent
# Or explicit legacy pipeline:
python3 espresso_agent.py --date 2026-05-18 --mode deterministic
```

## Rendering

`render_edition.py` is the entry point. It chains two modules:

- **`render_html.py`** — converts edition JSON into the variant_c template used by AI Garage (`edition_N_variant_c.html` and `.md`). Embeds the sniffer-friendly meta tokens (`NO. NNN`, dateline, shot count, preheader) that `sync-espresso.mjs` reads.
- **`render_images.py`** — two-step illustrations per edition: (1) Claude/`pplx` writes a scene brief per story; (2) `bin/asi-generate-image` renders PNGs with `nano_banana_2` (16:9 for stories, 1:1 for the prompt card). The bundled CLI uses `GEMINI_API_KEY` when set, else `OPENAI_API_KEY`. Pillow placeholders only if both are missing.

Issue numbers are monotonic — the renderer scans `editions/` for the highest existing `edition_N_*` and increments.

## Cross-edition memory

`data/archive.jsonl` stores a fingerprint per shipped story. Scout reads this before each run and excludes anything covered in the dedupe window, preventing the same story from shipping twice across editions.

## Observability — the agent trace

Every edition includes a full `agent_trace` array. Each event:

```json
{
  "ts": "2026-05-21T13:01:42Z",
  "role": "editor",
  "kind": "tool_call",
  "name": "pick",
  "args": { "id": 2, "slot": "...", "reason": "..." },
  "result_summary": "...",
  "thinking": "...",
  "tool_use_id": "toolu_..."
}
```

Working memory is appended as a final trace event and weak-pool notes flow into `edition.notes`.

A `notes` entry of `"mode: agent"` confirms the agent path completed; `"mode: deterministic"` indicates fallback.

## Daily cron

`.github/workflows/daily-edition.yml` runs every day at `0 11 * * *` UTC — **7am ET during EDT, 6am ET during EST** (GitHub Actions does not observe DST):

1. Checks out the repo
2. Runs the agent to produce edition JSON
3. Runs the renderer to produce HTML, Markdown, and four illustrations
4. Sends the edition as an HTML email with inline images
5. Commits everything to `main`

Required repo secrets:

| Secret | Purpose |
|---|---|
| `ANTHROPIC_API_KEY` | LLM calls (Claude) for ranking, drafting, and image scene briefs |
| `PERPLEXITY_API_KEY` | **Recommended** — `search_news` via Perplexity Search API (production path on Python 3.11; `pplx-cli` is optional on 3.12+ only) |
| `GEMINI_API_KEY` | **Illustrations (preferred)** — Nano Banana 2 (`gemini-3.1-flash-image-preview`). Without this, falls back to `OPENAI_API_KEY` / gpt-image-1. |
| `OPENAI_API_KEY` | Illustration fallback when Gemini is not set |
| `AI_ESPRESSO_FROM` | Sender address, e.g. `you@gmail.com` |
| `AI_ESPRESSO_TO`   | Comma-separated recipients, e.g. `you@work.com` |
| `GMAIL_APP_PASSWORD` | 16-char Gmail [app password](https://myaccount.google.com/apppasswords) used for SMTP |
| `SLACK_WEBHOOK_URL` | Optional — when set, `notify-failure` posts workflow failures to Slack; if unset, Slack step is skipped |

The workflow has `permissions: contents: write` so it can push commits. To run without sending the email, dispatch the workflow manually with `skip_email=true`.

## AI Garage integration

The portal in [`jackiehimel/ai-garage`](https://github.com/jackiehimel/ai-garage) already has a sync script (`scripts/sync-espresso.mjs`) and a workflow (`.github/workflows/sync-espresso.yml`) that:

1. Clones this repo
2. Finds the highest-numbered `editions/edition_N_variant_c.{html,md}`
3. Copies it to `editions/latest.html` + `editions/latest.md`
4. Copies referenced image assets to the same relative paths
5. Updates `editions/manifest.json` with the new issue
6. Commits to the AI Garage repo, which redeploys on Vercel

To switch AI Garage from weekly to daily syncs, edit one line in `ai-garage/.github/workflows/sync-espresso.yml`:

```yaml
# from
schedule:
  - cron: "0 14 * * 1"   # Mondays 14:00 UTC
# to
schedule:
  - cron: "0 12 * * *"   # daily 12:00 UTC (one hour after our 11:00 UTC publish)
```

No other changes to AI Garage are needed.

## Configuration

| File | Purpose |
|---|---|
| `sources.yaml` | Source catalog (tier, URL, enabled flag) — sole runtime config |
| `requirements.txt` | Python dependencies |

LLM calls route through the official Anthropic Python SDK (`call_llm_json` in `espresso_agent.py`). Override the model with the `ESPRESSO_MODEL` env var if you want to pin a specific snapshot; the default is `claude-sonnet-4-5`.

## Email delivery

`send_email.py` reads the rendered `edition_N_variant_c.html`, rewrites the four `<img src>` tags to `cid:` references, and ships a multipart/alternative email with inline PNG attachments. Transport is SMTP over `smtp.gmail.com:587` with STARTTLS, authenticated using a Gmail app password — Gmail signs the message with DKIM on the way out, so Microsoft 365 recipients accept it without further DNS setup.

Local dry run (no SMTP call, just prints the plan):

```bash
cd agent
AI_ESPRESSO_FROM=you@gmail.com \
AI_ESPRESSO_TO=you@work.com \
AI_ESPRESSO_DRY_RUN=1 \
python send_email.py ../editions/edition_1_variant_c.html ../editions/edition_1_variant_c.md
```
