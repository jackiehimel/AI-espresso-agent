# AI Espresso

A daily AI news digest generated by a native tool-use LLM agent. Each edition contains 3-6 story cards and one prompt-of-the-day card, rendered as HTML + Markdown and published to the AI Garage portal.

## Quick start

```bash
cd agent
pip install -r requirements.txt
export ANTHROPIC_API_KEY=sk-ant-...

# 1. Generate today's edition JSON (agent run)
python3 -c "import datetime as dt, espresso_agent; \
  espresso_agent.run(dt.date.today(), dry_run=False, use_cache=True, mode='agent')"

# 2. Render to HTML + Markdown + illustrations (optional: --no-images)
python3 render_edition.py $(date +%Y-%m-%d)

# 3. Preview in browser (agent → render → local server; Ctrl+C to stop)
python3 preview_edition.py $(date +%Y-%m-%d) --use-cache
python3 preview_edition.py 2026-05-18 --render-only   # re-render + open existing JSON
python3 preview_edition.py 2026-05-18 --no-images     # faster, broken image icons OK
python3 preview_edition.py 2026-05-18 --write-archive # opt in to archive mutation
```

`preview_edition.py` skips archive writes by default (sets `ESPRESSO_SKIP_ARCHIVE=1` during preview runs) so local previews do not mutate cross-edition memory.

Output:
- `data/editions/YYYY-MM-DD.json` — raw edition data + agent trace
- `../editions/edition_N_variant_c.html` — rendered email-ready HTML
- `../editions/edition_N_variant_c.md` — same content as Markdown
- `../editions/edition_N/assets/variant_c_NN.png` — one illustration per card

The `editions/` directory at the repo root is the contract surface that the AI Garage portal consumes via its existing `sync-espresso.mjs` script.

### Dev backfill (multiple dates)

There is no `run_chain.py` in this repo. To backfill several consecutive editions locally (so each day's `archive.jsonl` entry feeds the next run's dedupe), loop `espresso_agent.run` yourself:

```bash
cd agent
for d in 2026-05-16 2026-05-17 2026-05-18; do
  python3 -c "import datetime as dt, espresso_agent; \
    espresso_agent.run(dt.date.fromisoformat('$d'), dry_run=False, use_cache=True, mode='agent')"
  python3 render_edition.py "$d"
done
```

Not used by CI — production uses `.github/workflows/daily-edition.yml` only.

## Architecture

```
RSS fetch (sources.yaml) ─┐
                          ├─▶ Ranker (one LLM call, all candidates)
Perplexity discovery ─────┘            │
                                       ▼
                          pre-fetch article bodies (top 20)
                                       │
                                       ▼
                    ┌──────────────────────────────────┐
                    │  Editor (native tool_use loop)    │
                    │  pick · unpick · read_candidate   │
                    │  search_news · ship_edition       │
                    └──────────────────────────────────┘
                                       │  deterministic ship gates
                                       ▼
                                 edition JSON
                                       │
                                       ▼
                              render_edition.py
                                       │
                                       ▼
                        variant_c HTML + MD + PNGs
                                       │
                                       ▼
                            committed to editions/
                                       │
                                       ▼
                          AI Garage sync picks it up
```

### Sources

Candidate stories come from a tiered catalog (see `sources.yaml`):

| Tier | Examples |
|---|---|
| T1 — Frontier press & primary | Anthropic, OpenAI, DeepMind, NYT Tech, The Verge, TechCrunch, Bloomberg, arXiv cs.AI |
| T2 — Expert commentary | Stratechery, Latent Space, Simon Willison, Pragmatic Engineer, Ethan Mollick |
| T3 — Newsletters | TLDR AI, Rundown AI, Superhuman |
| T4 — Industry verticals | MIT Tech Review, FierceHealthcare, STAT |

### Ranking + discovery (`espresso_loop.py` — `rank_headlines()`, `discover_via_search()`)

The full candidate pool (RSS fetch plus Perplexity discovery results, filtered through the search-domain allow-list) is scored in a single LLM call against the editorial rubric — no lossy pre-funnel. The ranker returns the top 20 with a 0-100 score, a persona tag (`market` / `everyday` / `build` / `industry`, rendering labels only), and explicit coverage gaps. Article bodies for the top 20 are then pre-fetched in plain HTTP (no LLM cost) so the Editor starts with verified content.

### `search_news` backend

`search_news` uses the [Perplexity Search API](https://docs.perplexity.ai/docs/search/quickstart) (`POST /search`) via **`PERPLEXITY_API_KEY`** (or `PPLX_API_KEY`). Results are normalized to `title` / `url` / `snippet` / `domain` and filtered through the source allow-list. If the API is unavailable, the tool returns `{"error": "search failed — …"}` (never silent) and the trace records it so the Editor knows search was down.

### Editor (`espresso_loop.py` — `_EDITOR_SYSTEM`, `_run_editor_loop()`)

Runs an Anthropic native `tool_use` loop. The model chooses tools and order; Python dispatches and enforces ship gates. Tools:

| Tool | Purpose |
|---|---|
| `pick(id, reason, persona)` | Select a candidate (rejected immediately if it has no verified body, repeats a recent edition, or breaks the 2-per-vendor cap) |
| `unpick(id)` | Remove a previously picked candidate |
| `read_candidate(id)` | Fetch the article body for a candidate outside the pre-fetched set |
| `search_news(query)` | Web search beyond the daily fetch (max 3 per edition) |
| `ship_edition()` | Terminal tool — runs the deterministic ship gates |

### Ship gates

`ship_edition` succeeds only when all deterministic gates pass:

- 3+ picks (max 6), each with a verified article body
- at least one tier-1 source
- constitution checks pass (`constitution.py`), with at most one non-load-bearing AI story
- **cross-edition dedupe rails** (`dedupe_gate.py`): no pick matches a recently shipped story by canonical URL, normalized title, or embedding similarity

Gate failures return as errors into the tool loop, so the Editor unpicks and chooses different news — the gates are rails, not overrides. If the tool budget is exhausted with 3+ valid picks, the run **force-ships** the slate (recorded in the trace). With fewer than 3 recoverable picks, the run writes `data/editions/YYYY-MM-DD.failed.json` and fails CI unless `ESPRESSO_ALLOW_DETERMINISTIC_FALLBACK=1` (local dev only).

### Controller (`espresso_loop.py` — `agentic_select()`)

Pool dedupe (fingerprints + exact archive match) → discovery merge → `rank_headlines()` → `prefetch_bodies()` → `_run_editor_loop()` (hard budget 20 tool calls). Records every event to `agent_trace` including `tool_use_id`. On unrecoverable agent failure, the run writes `data/editions/YYYY-MM-DD.failed.json` and **raises** (CI fails). It does **not** silently fall back unless you opt in locally (see below).

### Deterministic fallback (local dev emergency only)

Production and `.github/workflows/daily-edition.yml` always use `mode="agent"`. The legacy `rank_and_select` path (`RANKING_SYSTEM` in `espresso_agent.py`) is **deprecated** and is **not** kept in sync with `_EDITORIAL_RUBRIC` in `espresso_loop.py` — do not rely on it for editorial quality.

| Knob | Purpose |
|---|---|
| `mode="agent"` (default) | Rank → prefetch → Editor native `tool_use` loop |
| `mode="deterministic"` | Skip the agent loop; run `rank_and_select` directly (legacy) |
| `ESPRESSO_ALLOW_DETERMINISTIC_FALLBACK=1` | After agent failure with no recoverable slate, run `rank_and_select` instead of failing |
| `ESPRESSO_SKIP_ARCHIVE=1` | Dev/local only: write edition JSON but skip `append_archive` in `write_edition` (does not replace Phase 1.1 upsert-by-date) |

Use the env flag or `--mode deterministic` only on your machine when debugging source fetch or ranking plumbing — never in CI, cron, or client-facing runs. Example:

```bash
# Agent failed; inspect trace first — fallback is last resort:
ESPRESSO_ALLOW_DETERMINISTIC_FALLBACK=1 python3 espresso_agent.py --date 2026-05-18 --mode agent
# Local agent run without mutating archive.jsonl:
ESPRESSO_SKIP_ARCHIVE=1 python3 espresso_agent.py --date 2026-05-19 --use-cache --mode agent
# Or explicit legacy pipeline:
python3 espresso_agent.py --date 2026-05-18 --mode deterministic
```

## Rendering

`render_edition.py` is the entry point. It chains two modules:

- **`render_html.py`** — converts edition JSON into the variant_c template used by AI Garage (`edition_N_variant_c.html` and `.md`). Embeds the sniffer-friendly meta tokens (`NO. NNN`, dateline, shot count, preheader) that `sync-espresso.mjs` reads.
- **`render_images.py`** — two-step illustrations per edition: (1) Claude writes a scene brief per story; (2) `bin/asi-generate-image` renders PNGs with `nano_banana_2` (16:9 for stories, 1:1 for the prompt card). The bundled CLI uses `GEMINI_API_KEY` when set, else `OPENAI_API_KEY`. Pillow placeholders only if both are missing.

Issue numbers are monotonic — the renderer scans `editions/` for the highest existing `edition_N_*` and increments.

## Cross-edition memory + dedupe gate

`data/archive.jsonl` stores one row per shipped edition: story fingerprints, rewritten headlines, original headlines, and source URLs. Before each run the agent builds an `ArchiveIndex` from this archive, and `dedupe_gate.py` enforces three layers of repeat detection:

| Layer | Catches | Window |
|---|---|---|
| Canonical URL (query/fragment/`www` stripped) | The exact same article resurfacing | 30 days |
| Normalized title (lowercased, punctuation stripped) | Syndicated copies of the same headline | 30 days |
| Embedding similarity (Gemini `gemini-embedding-001`, cosine ≥ 0.80) | The same story re-reported under a different headline or source | 10 days |

The exact layers run at pool dedupe, at `pick` time, and again at `ship_edition` (search results bypass pool dedupe). The semantic layer runs at `ship_edition`. A repeat surfaces as a tool error telling the Editor to unpick and choose different news. The semantic layer fails open — if no embedding API key is configured or the API is down, the gate skips rather than blocking the daily edition.

## Observability — the agent trace

Every edition includes a full `agent_trace` array. Each event:

```json
{
  "ts": "2026-05-21T13:01:42Z",
  "role": "editor",
  "kind": "tool_call",
  "name": "pick",
  "args": { "id": 2, "reason": "...", "persona": "build" },
  "result_summary": "...",
  "tool_use_id": "toolu_..."
}
```

A `notes` entry of `"mode: agent"` confirms the agent path completed; `"mode: deterministic"` indicates fallback.

## Daily cron

`.github/workflows/daily-edition.yml` runs every day at `0 11 * * *` UTC — **7am ET during EDT, 6am ET during EST** (GitHub Actions does not observe DST):

1. Checks out the repo and runs the unit test suite
2. Checks the duplicate-send guard — if the edition for this date is already committed, generation and email are skipped (no resends, no API spend)
3. Runs the agent to produce edition JSON
4. Runs the renderer to produce HTML, Markdown, and illustrations
5. Sends the edition as an HTML email with inline images
6. Commits everything to `main`

Manual dispatch inputs: `date` (override edition date), `skip_email` (render only), `test_recipient` (send to a single preview address), and `force_resend` (bypass the duplicate-send guard when re-sending is intentional).

Required repo secrets:

| Secret | Purpose |
|---|---|
| `ANTHROPIC_API_KEY` | LLM calls (Claude) for ranking, the Editor loop, drafting, and image scene briefs |
| `PERPLEXITY_API_KEY` | Perplexity Search API — daily discovery merge plus the Editor's `search_news` tool |
| `GEMINI_API_KEY` | Embeddings for the semantic dedupe gate, and illustrations (preferred). Without it, illustrations fall back to `OPENAI_API_KEY` and the semantic gate skips. |
| `OPENAI_API_KEY` | Illustration fallback when Gemini is not set |
| `AI_ESPRESSO_FROM` | Sender address, e.g. `you@gmail.com` |
| `AI_ESPRESSO_TO`   | Comma-separated recipients, e.g. `you@work.com` |
| `GMAIL_APP_PASSWORD` | 16-char Gmail [app password](https://myaccount.google.com/apppasswords) used for SMTP |
| `SLACK_WEBHOOK_URL` | Optional — when set, `notify-failure` posts workflow failures to Slack; if unset, Slack step is skipped |

The workflow has `permissions: contents: write` so it can push commits.

## AI Garage integration

The portal in [`jackiehimel/ai-garage`](https://github.com/jackiehimel/ai-garage) already has a sync script (`scripts/sync-espresso.mjs`) and a workflow (`.github/workflows/sync-espresso.yml`) that:

1. Clones this repo
2. Finds the highest-numbered `editions/edition_N_variant_c.{html,md}`
3. Copies it to `editions/latest.html` + `editions/latest.md`
4. Copies referenced image assets to the same relative paths
5. Updates `editions/manifest.json` with the new issue
6. Commits to the AI Garage repo, which redeploys on Vercel

To switch AI Garage from weekly to daily syncs, edit one line in `ai-garage/.github/workflows/sync-espresso.yml`:

```yaml
# from
schedule:
  - cron: "0 14 * * 1"   # Mondays 14:00 UTC
# to
schedule:
  - cron: "0 12 * * *"   # daily 12:00 UTC (one hour after our 11:00 UTC publish)
```

No other changes to AI Garage are needed.

## Configuration

| File | Purpose |
|---|---|
| `sources.yaml` | Source catalog (tier, URL, enabled flag) — sole runtime config |
| `requirements.txt` | Python dependencies |

LLM calls route through the official Anthropic Python SDK (`call_llm_json` in `espresso_agent.py`). Override the model with the `ESPRESSO_MODEL` env var if you want to pin a specific snapshot; the default is `claude-sonnet-4-5`.

## Email delivery

`send_email.py` reads the rendered `edition_N_variant_c.html`, rewrites the `<img src>` tags to `cid:` references, and ships a multipart/alternative email with inline PNG attachments. It also enforces a hard no-resend guard: dates recorded in `data/sent_editions.json` are never re-sent unless `AI_ESPRESSO_FORCE_RESEND=1`. Transport is SMTP over `smtp.gmail.com:587` with STARTTLS, authenticated using a Gmail app password — Gmail signs the message with DKIM on the way out, so Microsoft 365 recipients accept it without further DNS setup.

Local dry run (no SMTP call, just prints the plan):

```bash
cd agent
AI_ESPRESSO_FROM=you@gmail.com \
AI_ESPRESSO_TO=you@work.com \
AI_ESPRESSO_DRY_RUN=1 \
python send_email.py ../editions/edition_1_variant_c.html ../editions/edition_1_variant_c.md
```
