# AI Espresso

A daily AI news digest generated by a multi-role LLM agent. Each edition contains three story cards and one prompt-of-the-day card, rendered as HTML + Markdown and published to the AI Garage portal.

## Quick start

```bash
cd agent
pip install -r requirements.txt
export ANTHROPIC_API_KEY=sk-ant-...

# 1. Generate today's edition JSON (agent run)
python3 -c "import datetime as dt, espresso_agent; \
  espresso_agent.run(dt.date.today(), dry_run=False, use_cache=True, mode='agent')"

# 2. Render to HTML + Markdown + illustrations (optional: --no-images)
python3 render_edition.py $(date +%Y-%m-%d)

# 3. Preview in browser (agent ŌåÆ render ŌåÆ local server; Ctrl+C to stop)
python3 preview_edition.py $(date +%Y-%m-%d) --use-cache
python3 preview_edition.py 2026-05-18 --render-only   # re-render + open existing JSON
python3 preview_edition.py 2026-05-18 --no-images     # faster, broken image icons OK
```

Output:
- `data/editions/YYYY-MM-DD.json` ŌĆö raw edition data + agent trace
- `../editions/edition_N_variant_c.html` ŌĆö rendered email-ready HTML
- `../editions/edition_N_variant_c.md` ŌĆö same content as Markdown
- `../editions/edition_N/assets/variant_c_NN.png` ŌĆö four illustrations

The `editions/` directory at the repo root is the contract surface that the AI Garage portal consumes via its existing `sync-espresso.mjs` script.

### Dev backfill (`run_chain.py`)

Runs `espresso_agent.run(date, mode="agent")` for a hard-coded list of dates (see the script). Use when you need several consecutive editions locally so each day's `archive.jsonl` entry feeds the next run's dedupe. Not used by CI ŌĆö production uses `.github/workflows/daily-edition.yml` only.

### Scout (`espresso_loop.py` ŌĆö `SCOUT_SYSTEM`, `run_scout()`)

Pulls candidate stories from a 45-source catalog (see `sources.yaml`) organized into four tiers:

| Tier | Examples |
|---|---|
| T1 ŌĆö Frontier press & primary | Anthropic, OpenAI, DeepMind, NYT Tech, The Verge, TechCrunch, Bloomberg, arXiv cs.AI |
| T2 ŌĆö Expert commentary | Stratechery, Latent Space, Simon Willison, Pragmatic Engineer, Ethan Mollick |
| T3 ŌĆö Newsletters | TLDR AI, Rundown AI, Superhuman |
| T4 ŌĆö Industry verticals | MIT Tech Review, FierceHealthcare, STAT |

Scout scores candidates on freshness, source tier, and topical fit, then hands the Editor a shortlist of ~14 stories along with explicit coverage gaps. The Editor uses these gaps to decide when to call `search_news`.

### `search_news` backends

Production (CI and local on Python 3.11) should use **`PERPLEXITY_API_KEY`** ŌĆö the [Perplexity Search API](https://docs.perplexity.ai/docs/search/quickstart) (`POST /search`) returns `results`, normalized to the same hit shape the agent expects (`title`, `url`, `snippet`, `domain`) before allow-list filtering.

| Backend | When | Requirement |
|---|---|---|
| **Search API** (recommended) | `PERPLEXITY_API_KEY` or `PPLX_API_KEY` set | Python 3.11+, `httpx` (already in `requirements.txt`) |
| **pplx CLI** (optional) | `pplx` on `PATH` and CLI succeeds | `pip install pplx-cli` on **Python 3.12+** only |

Order: try `pplx search web` first if the binary exists; on failure or absence, call the API. If both are unavailable, the tool returns `{"error": "search failed ŌĆö ŌĆ”"}` (never silent) and the trace records it in `result_summary` so the Editor can note search was unavailable.

### Editor (`espresso_loop.py` ŌĆö `EDITOR_SYSTEM`, `run_tool_agent()`)

Runs an Anthropic native `tool_use` loop. The model chooses tools and order; Python dispatches and enforces ship gates. Tools:

| Tool | Purpose |
|---|---|
| `pick(slot, id, reason)` | Assign a candidate to a slot |
| `unpick(slot)` | Remove the current pick from a slot |
| `read_candidate(id)` | Read the full body of a shortlist item |
| `search_news(query)` | Web search beyond the daily fetch (max 2 per edition). See **search_news backends** above ŌĆö prefer `PERPLEXITY_API_KEY` in production |
| `check_archive(headline)` | Fuzzy dedupe against recent editions |
| `update_memory(key, value)` | Write working memory (pool quality, gaps, notes) |
| `note_weak_pool(reason, adjustments)` | Document a thin news day (required for 2-story editions) |
| `self_critique()` | Rubric review ŌĆö returns `approve` or `revise` |
| `ship_edition()` | Terminal tool when gates pass |

### Ship gates

`ship_edition` succeeds only when: required picks are filled (3 normally, 2 on documented weak-pool days), tier-1 minimum is met, vendor cap respected, constitution checks pass (`constitution.py`), and the last `self_critique` was `approve`. If the Critic approves but `ship_edition` is not called (budget exhaust), the run **salvages** the approved slate (`mode: agent`, warning in notes). With zero recoverable picks, the run writes `data/editions/YYYY-MM-DD.failed.json` and fails CI unless `ESPRESSO_ALLOW_DETERMINISTIC_FALLBACK=1` (local dev only).

### Controller (`espresso_loop.py` ŌĆö `agentic_select()`)

Scout bootstrap ŌåÆ `run_tool_agent()` (soft budget 25, hard budget 40 tool calls). Records every event to `agent_trace` including `tool_use_id`. Falls back to deterministic `rank_and_select` on LLM errors or failed ship.

## Rendering

`render_edition.py` is the entry point. It chains two modules:

- **`render_html.py`** ŌĆö converts edition JSON into the variant_c template used by AI Garage (`edition_N_variant_c.html` and `.md`). Embeds the sniffer-friendly meta tokens (`NO. NNN`, dateline, shot count, preheader) that `sync-espresso.mjs` reads.
- **`render_images.py`** ŌĆö two-step illustrations per edition: (1) Claude/`pplx` writes a scene brief per story; (2) `bin/asi-generate-image` renders PNGs with `nano_banana_2` (16:9 for stories, 1:1 for the prompt card). The bundled CLI uses `GEMINI_API_KEY` when set, else `OPENAI_API_KEY`. Pillow placeholders only if both are missing.

Issue numbers are monotonic ŌĆö the renderer scans `editions/` for the highest existing `edition_N_*` and increments.

## Cross-edition memory

`data/archive.jsonl` stores a fingerprint per shipped story. Scout reads this before each run and excludes anything covered in the dedupe window, preventing the same story from shipping twice across editions.

## Observability ŌĆö the agent trace

Every edition includes a full `agent_trace` array. Each event:

```json
{
  "ts": "2026-05-21T13:01:42Z",
  "role": "editor",
  "kind": "tool_call",
  "name": "pick",
  "args": { "id": 2, "slot": "...", "reason": "..." },
  "result_summary": "...",
  "thinking": "...",
  "tool_use_id": "toolu_..."
}
```

Working memory is appended as a final trace event and weak-pool notes flow into `edition.notes`.

A `notes` entry of `"mode: agent"` confirms the agent path completed; `"mode: deterministic"` indicates fallback.

## Daily cron

`.github/workflows/daily-edition.yml` runs every day at `0 11 * * *` UTC ŌĆö **7am ET during EDT, 6am ET during EST** (GitHub Actions does not observe DST):

1. Checks out the repo
2. Runs the agent to produce edition JSON
3. Runs the renderer to produce HTML, Markdown, and four illustrations
4. Sends the edition as an HTML email with inline images
5. Commits everything to `main`

Required repo secrets:

| Secret | Purpose |
|---|---|
| `ANTHROPIC_API_KEY` | LLM calls (Claude) for ranking, drafting, and image scene briefs |
| `PERPLEXITY_API_KEY` | **Recommended** ŌĆö `search_news` via Perplexity Search API (production path on Python 3.11; `pplx-cli` is optional on 3.12+ only) |
| `GEMINI_API_KEY` | **Illustrations (preferred)** ŌĆö Nano Banana 2 (`gemini-3.1-flash-image-preview`). Without this, falls back to `OPENAI_API_KEY` / gpt-image-1. |
| `OPENAI_API_KEY` | Illustration fallback when Gemini is not set |
| `AI_ESPRESSO_FROM` | Sender address, e.g. `you@gmail.com` |
| `AI_ESPRESSO_TO`   | Comma-separated recipients, e.g. `you@work.com` |
| `GMAIL_APP_PASSWORD` | 16-char Gmail [app password](https://myaccount.google.com/apppasswords) used for SMTP |

The workflow has `permissions: contents: write` so it can push commits. To run without sending the email, dispatch the workflow manually with `skip_email=true`.

## AI Garage integration

The portal in [`jackiehimel/ai-garage`](https://github.com/jackiehimel/ai-garage) already has a sync script (`scripts/sync-espresso.mjs`) and a workflow (`.github/workflows/sync-espresso.yml`) that:

1. Clones this repo
2. Finds the highest-numbered `editions/edition_N_variant_c.{html,md}`
3. Copies it to `editions/latest.html` + `editions/latest.md`
4. Copies referenced image assets to the same relative paths
5. Updates `editions/manifest.json` with the new issue
6. Commits to the AI Garage repo, which redeploys on Vercel

To switch AI Garage from weekly to daily syncs, edit one line in `ai-garage/.github/workflows/sync-espresso.yml`:

```yaml
# from
schedule:
  - cron: "0 14 * * 1"   # Mondays 14:00 UTC
# to
schedule:
  - cron: "0 12 * * *"   # daily 12:00 UTC (one hour after our 11:00 UTC publish)
```

No other changes to AI Garage are needed.

## Configuration

| File | Purpose |
|---|---|
| `sources.yaml` | Source catalog (tier, URL, enabled flag) ŌĆö sole runtime config |
| `requirements.txt` | Python dependencies |

LLM calls route through the official Anthropic Python SDK (`call_llm_json` in `espresso_agent.py`). Override the model with the `ESPRESSO_MODEL` env var if you want to pin a specific snapshot; the default is `claude-sonnet-4-5`.

## Email delivery

`send_email.py` reads the rendered `edition_N_variant_c.html`, rewrites the four `<img src>` tags to `cid:` references, and ships a multipart/alternative email with inline PNG attachments. Transport is SMTP over `smtp.gmail.com:587` with STARTTLS, authenticated using a Gmail app password ŌĆö Gmail signs the message with DKIM on the way out, so Microsoft 365 recipients accept it without further DNS setup.

Local dry run (no SMTP call, just prints the plan):

```bash
cd agent
AI_ESPRESSO_FROM=you@gmail.com \
AI_ESPRESSO_TO=you@work.com \
AI_ESPRESSO_DRY_RUN=1 \
python send_email.py ../editions/edition_1_variant_c.html ../editions/edition_1_variant_c.md
```
